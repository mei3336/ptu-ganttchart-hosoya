### 0.1 基本設計書からの変更・補足点

基本設計書11章は「ヒアリング内容とソースコード確認から推測した項目一覧」であり、フィールドの過不足が明記されていた。本書ではソースコード（`ganttforge.html`）を追加確認し、以下を補足・確定した。

| ストア | 補足内容 | 根拠 |
|---|---|---|
| `tasks`（旧`todos`） | `projectId` フィールドが必須であることを追加（基本設計書11.3には記載漏れ）。プロジェクトごとにタスク管理パネルを開くため、`projectId`が無いと絞り込みができない | `dbGetIdx('todos','projectId',S.currentId)` |
| `tasks`（旧`todos`） | `status`は自由文字列ではなく `backlog`/`doing`/`done` の3値固定（カンバン列に対応） | `KB_COLS`定数 |
| `tasks`（旧`todos`） | `order`フィールドを追加（カンバン内の並び順、ドラッグ&ドロップで更新） | `submitTodo` / `dropTodoCard` |
| `changelog` | `projectId`フィールドを追加（基本設計書には記載なし）。プロジェクト未選択時は`'global'`が入る | `writeLog`関数、`dbGetIdx('changelog','projectId',...)` |
| `memos` | `projectId`・`order`フィールドを追加（基本設計書11.6には記載なし）。基本設計書では「アプリ全体で共通」と誤って推測していたが、実際は他ストアと同様にプロジェクトごとに切り替わる | `createMemo`関数、`dbGetIdx('memos','projectId',...)` |

これ以外は基本設計書11章の内容をそのまま採用する。

## 1. 設計方針（全ストア共通）

| 方針 | 内容 | 理由 |
|---|---|---|
| 主キー | 全ストア `id`（文字列） | エクスポート/インポート・スナップショット復元・変更履歴の「戻す」機能で、同一レコードをアプリ側で発行したIDのまま複数ストア間・複数タイミングで扱う必要があるため、DB任せの`autoIncrement`ではなくアプリ側でID発行する方式にする |
| ID生成方法 | アプリ側で一意な文字列を発行（例：タイムスタンプ+ランダム文字列） | `autoIncrement`は同一データベース内でのみ一意性を保証するため、JSONインポート時に他環境で発行されたIDと衝突しない設計にする必要がある |
| 日付の型 | `startDate`/`endDate`/`dueDate`等は `"YYYY-MM-DD"` の文字列 | ガントチャートの日付比較・カレンダーinput要素とのやり取りをシンプルにするため（Dateオブジェクトのシリアライズ問題を避ける） |
| 日時の型 | `createdAt`等は ISO 8601文字列（例：`"2026-07-14T10:00:00.000Z"`） | ソート（文字列比較で時系列順になる）と変更履歴表示のしやすさを優先 |
| インデックスの基準 | 「一覧画面を開くときに絞り込みで使うフィールド」にのみ`index`を張る | 単純な`id`取得や全件取得（`getAll`）にはインデックス不要。インデックスは検索コストとのトレードオフのため、実際に絞り込みが発生する項目に限定する |

## 2. ストア間の関連図

```
projects (1) ──< schedules (旧tasks)         : projectId
projects (1) ──< tasks (旧todos)             : projectId
projects (1) ──< issues                      : projectId
projects (1) ──< quickmemos                  : projectId
projects (1) ──< snapshots                   : projectId
projects (1) ──< changelog                   : projectId（未選択時は 'global'）
schedules (1) ─< schedules                   : parentId（自己参照・階層構造）
schedules (1) ─< comments                    : taskId（スケジュールIDを指す）
issues (1) ──< issues                        : parentNodeId（自己参照・階層構造）
projects (1) ──< memos                       : projectId
```

- `memos`は他のプロジェクト付随ストアと同様に`projectId`を持ち、プロジェクトごとにメモ一覧が切り替わる（ソースコード確認により訂正。当初「アプリ全体共通」と記載していたのは誤りだった）
- `comments`は`projectId`を持たず、`taskId`（スケジュールID）経由でプロジェクトに間接的に紐づく（コメント一覧パネルは、現在のプロジェクトのスケジュールID集合でコメントを絞り込む実装のため）

## 3. ストア定義一覧

### 3.1 projects

| 項目 | 内容 |
|---|---|
| ストア名 | `projects` |
| keyPath | `id` |
| インデックス | なし |

**インデックス不要の理由**：プロジェクト選択ドロップダウンは常に全件（`getAll`）を表示するため、絞り込み用インデックスが不要。

| フィールド名 | 型 | 必須 | 内容 |
|---|---|---|---|
| id | string | ○ | 主キー |
| name | string | ○ | プロジェクト名 |
| startDate | string (date) | ○ | 開始日 |
| endDate | string (date) | ○ | 終了日（バリデーションなし。開始日より前でも保存可＝既知の非整合、忠実再現） |
| milestones | array\<{name: string, date: string}\> | ○（空配列可） | マイルストーン一覧 |
| locked | boolean | ○ | ロック状態 |
| createdAt | string (datetime) | ○ | 作成日時 |

### 3.2 schedules（旧`tasks`）

| 項目 | 内容 |
|---|---|
| ストア名 | `schedules` |
| keyPath | `id` |
| インデックス | `projectId`（非unique）／`parentId`（非unique） |

**インデックス理由**：
- `projectId`：プロジェクト切替のたびに「このプロジェクトのスケジュール一覧」を取得する必要があるため
- `parentId`：階層構造で「この行の子スケジュール一覧」を取得し、子の日付から親の日付を再計算する処理（`syncAncestorDates`相当）で使用するため

| フィールド名 | 型 | 必須 | 内容 |
|---|---|---|---|
| id | string | ○ | 主キー |
| projectId | string | ○ | 所属プロジェクトID（インデックス対象） |
| parentId | string \| null | - | 親スケジュールID（インデックス対象。ルート行は`null`） |
| name | string | ○ | スケジュール名 |
| startDate | string (date) | - | 開始日 |
| endDate | string (date) | - | 終了日（終了日<開始日はバリデーションエラーとする。プロジェクトとの非対称仕様は忠実再現） |
| assignee | string | - | 担当者 |
| taskStatus | "todo" \| "inprogress" \| "done" | ○ | ステータス（未着手／進行中／完了） |
| notes | string | - | メモ |
| color | string (例: "#2563EB") | - | カラー |
| order | number | ○ | 並び順 |

### 3.3 tasks（旧`todos`）

| 項目 | 内容 |
|---|---|
| ストア名 | `tasks` |
| keyPath | `id` |
| インデックス | `projectId`（非unique） |

**インデックス理由**：タスク管理パネル（カンバン）はプロジェクトごとに開くため、`dbGetIdx`相当で現在のプロジェクトのタスクのみ取得する必要がある。

| フィールド名 | 型 | 必須 | 内容 |
|---|---|---|---|
| id | string | ○ | 主キー |
| projectId | string | ○ | 所属プロジェクトID（インデックス対象。基本設計書からの補足） |
| title | string | ○ | タイトル |
| status | "backlog" \| "doing" \| "done" | ○ | カンバンの列（基本設計書からの補足：自由文字列ではなく3値固定） |
| done | boolean | ○ | 完了フラグ（`status==='done'`と同期） |
| doneAt | string (datetime) \| null | - | 完了日時 |
| priority | 1 \| 2 \| 3 \| 4 | ○ | 優先度（1:高 〜 4:なし） |
| dueDate | string (date) \| "" | - | 期日 |
| description | string | - | 詳細メモ |
| order | number | ○ | カンバン内の並び順（基本設計書からの補足） |

※スケジュール（`schedules.taskStatus`）とは連携しない仕様（既知の非整合。基本設計書1.3-No.2）。データモデル上も両ストアは独立し、双方向の外部キーは持たせない。

### 3.5 comments

| 項目 | 内容 |
|---|---|
| ストア名 | `comments` |
| keyPath | `id` |
| インデックス | `taskId`（スケジュールIDを指す。非unique） |

**インデックス理由**：コメント一覧パネル・ガント上の個別スケジュールの吹き出しで「特定スケジュールに紐づくコメント」を取得するため。

**`projectId`を持たせない理由**：コメントはスケジュール単位の会話ログであり、所属プロジェクトは`taskId`経由（`schedules.projectId`）で辿れるため、正規化の観点から重複保持しない。コメント一覧パネルは、現在のプロジェクトの全スケジュールID集合を先に取得し、それに一致するコメントを絞り込む実装とする。

| フィールド名 | 型 | 必須 | 内容 |
|---|---|---|---|
| id | string | ○ | 主キー |
| taskId | string | ○ | 紐づくスケジュールID（`schedules.id`。インデックス対象） |
| text | string | ○ | コメント内容 |
| createdAt | string (datetime) | ○ | 投稿日時 |

### 3.6 memos

| 項目 | 内容 |
|---|---|
| ストア名 | `memos` |
| keyPath | `id` |
| インデックス | `projectId`（非unique） |

**訂正**：当初「プロジェクトに属さないアプリ全体共通のストア」としていたが誤り。ソースコード確認の結果、`memos`は他ストアと同様に`projectId`を持ち、プロジェクトを切り替えるとメモ一覧も切り替わる仕様だった。

**インデックス理由**：メモパネルは「現在選択中のプロジェクトのメモのみ」を一覧表示するため、`projectId`による絞り込みが必要。

| id | string | ○ | 主キー |
| projectId | string | ○ | 所属プロジェクトID（インデックス対象。当初の基本設計書には記載がなかったが、ソースコード確認により追加） |
| title | string | ○ | メモタイトル |
| body | string (Markdown) | - | 本文 |
| order | number | ○ | メモ一覧内の並び順（当初の基本設計書には記載がなかったが、ソースコード確認により追加） |
| createdAt | string (datetime) | ○ | 作成日時 |
| updatedAt | string (datetime) | ○ | 更新日時 |

### 3.9 changelog（変更履歴）

| 項目 | 内容 |
|---|---|
| ストア名 | `changelog` |
| keyPath | `id` |
| インデックス | `projectId`（非unique） |

**インデックス理由**：変更履歴パネルはプロジェクトごとに表示するため（プロジェクト未選択時の操作は`projectId:"global"`として記録し、絞り込みの対象外とする）。

| フィールド名 | 型 | 必須 | 内容 |
|---|---|---|---|
| id | string | ○ | 主キー |
| projectId | string | ○ | 所属プロジェクトID（インデックス対象。プロジェクト未選択時は`"global"`。基本設計書からの補足） |
| action | "add" \| "edit" \| "delete" | ○ | 操作種別 |
| store | string（例："projects","schedules","tasks","issues"） | ○ | 対象ストア名 |
| storeLabel | string（例："プロジェクト","スケジュール"） | ○ | 対象ストアの日本語表示名 |
| itemId | string | ○ | 対象レコードのID |
| itemName | string | ○ | 対象レコードの名称（表示用） |
| changes | array\<{field, label, oldVal, newVal}\> | ○（空配列可） | フィールド差分（新規作成時は空配列） |
| snapshot | object \| null | - | 変更前の状態（「戻す」機能で使用。新規作成時は`null`） |
| createdAt | string (datetime) | ○ | 記録日時 |

※対象は`projects` / `schedules` / `tasks` / `issues`の4ストアのみ（`comments`・`memos`・`quickmemos`・`snapshots`は変更履歴の対象外）。

※「戻す」操作は`snapshot`をレコード1件分書き戻すのみで、親子関係の整合性は保証しない（既知の非整合。基本設計書1.3-No.3）。データモデル上も、変更履歴の巻き戻し時に子レコードの存在確認処理は行わない前提とする。

## 6. 非機能要件・実装時の注意点との対応

- 「終了日<開始日」の非対称バリデーション（基本設計書1.3-No.1）は、データモデルの型やインデックスには影響しない。保存前のアプリケーション層（フォーム送信処理）でのみ差を設ける
- 変更履歴の対象ストアが `projects` / `schedules` / `tasks` / `issues` の4つに限定される点は、実装時に対象ストアの集合を定数（例：`LOGGED_STORES`）としてまとめ、命名規約に沿った分かりやすい名前を付ける
- IndexedDBのトランザクション内で他の非同期処理を待たない方針（`CLAUDE.md`）のため、複数ストアにまたがる書き込み（例：プロジェクト削除時の配下スケジュール一括削除）は、事前に`getAll`等でデータを読み切ってから、書き込み用トランザクションを開始する設計とする
