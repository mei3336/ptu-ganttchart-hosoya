# 差分発生原因分析 ＆ AI共同開発プロセス改善報告書 (完全版)

本報告書は、研修再現フェーズにおける見本アプリ（`ganttforge.html`）と、12時時点の実装ファイル（細谷作 `index.html`）との間に発生した**UI・機能・ロジックのあらゆる「ズレ（差分）」の発生原因を徹底的に自己分析し、AI（Claude Code）との共同開発プロセスにおける自身の工夫、こだわり、および「プロ意識の不足」について、愛のある厳しい視点で振り返った研修の最終成果物**です。

## 1. AI協働プロセスの振り返り（行ってよかったこと vs 不足していたこと）

振り返りソース①「テストファースト設計術」および②「UX言語化の技術」の思想に基づき、今回の開発プロセスにおける自身の行動を自己評価として客観的に整理しました。

| 評価区分 | 具体的な行動・プロセス | 得られた成果 / 発生した問題 | 振り返りソースとの合致・本質的理由 |

| **行ってよかったこと<br>（Good）** | **1. お手本のコードを一切コピペ・流用せず、日本語の言語化のみでAIにUXの指示を出した** | AIが単なる「お手本コードのコピペマシーン」にならず、自分たちの頭で設計構造（IndexedDBへの非同期移行など）を考える余地を残した。 | **「UX言語化の技術」§1 に合致。**<br>実際の現場にはお手本コードは存在しないため、ソースを見せずに自力で言語化を試みたアプローチ自体は100%大正解。 |

## 2. 差分発生原因分析：なぜお手本と実装にこれほどのズレが生じたのか？

12時時点のコードに発生した微細なUIズレ、大量のスタブ、および「名残りコード」の発生原因を、1行レベルの「生の事実」から精密に分析します。

### 2.1. UIデザイン・手触りの精密なズレ

#### ① モーダル共通のアニメーションと角丸の喪失（UI No.31, 32）
*   **生の差分**: 
    *   お手本：`.modal-panel` の角丸は14px、パディングは28px、表示時はふわっと浮き上がる `@keyframes slideUp` アニメーションが走り、背景（`.modal-overlay`）は `backdrop-filter: blur(4px)` でぼかされる。
    *   実ファイル：角丸8px、ぼかしなし、アニメーションなしで「パッと無機質に表示される」。
*   **発生原因**:
    Meiさんが指示を出す際に「なめらかにふわっと開くモーダル」と日本語で抽象的に伝えたため、Claudeが具体的なCSSキーフレームや `backdrop-filter` の記述を「不要な演出」とみなしてすべて省略し、標準トークンの角丸8pxの静的コンテナとしてのみ実装したためです。

#### ② フォーカス時の青い枠線（2重グロー効果）の欠落（UI No.29）
*   **生の差分**:
    *   お手本：入力欄（`.fi`）やセレクトボックスクリック時、`border-color: var(--accent); box-shadow: 0 0 0 3px rgba(37,99,235,.1)` が当たり、操作箇所が明確になる。
    *   実ファイル：フォーカス時の疑似クラス（`:focus-visible` 等）の定義がモーダル内 input 要素に存在しない。
*   **発生原因**:
    「ホバーやアクティブ時の手触りルール（デザイントークン）」を、開発の最初にClaudeにシステム制約として握らせていなかったため、AIがブラウザのデフォルト挙動（地味な黒い外枠）に処理を任せて手抜き実装をスルーしたことが原因です。

#### ③ カラースウォッチ（カラーパレット）の形状・サイズ不整合（UI No.30）
*   **生の差分**:
    *   お手本：11色パレットは `width:28px;height:28px;border-radius:7px` の「角丸四角形」で、選択時は `transform:scale(1.18)` で拡大される。
    *   実ファイル：サイズ24px、`border-radius:50%` の「完全な円形」、選択時は単なる黒枠線になる。
*   **発生原因**:
    「色の選択パレット」という大雑把な言葉でClaudeに指示したため、AIが世の中に溢れている一般的な円形パレットのコードを学習データからそのまま出力し、お手本の「角丸四角形 ＋ 拡大アニメーション」という固有のデザインUXを勝手に無視したためです。

### 2.2. 機能スタブの大量残存（妥協の残骸）

#### ① 祝日判定背景（dayType）と営業日数算出（bizDays）のスタブ化（UI No.45 / JS No.34, 35）
*   **生の差分**:
    *   お手本：日本の祝日マップ（`HOLIDAYS` の Set）を基にカレンダーのグリッド背景をオレンジ系（`rgba(234,88,12,.12)`）に色分けし、営業日数（土日・祝日を除く）を正確に逆算する。
    *   実ファイル：祝日判定および営業日算出ロジックは完全未実装（空ファイル）。カレンダー背景も土日しか色分けされない。
*   **発生原因**:
    詳細設計書に「営業日数算出は実装予定」と曖昧なスコープが残っていたため、細谷さんとAIが「今作らなくても怒られない、後回しにしよう」と勝手に判断し、チェックリスト（観点表）による進捗監査がなかったために完全にスルーされました。

#### ② Excel・PDF出力（exportExcel / exportPDF）の未実装スタブ（JS No.142）
*   **生の差分**:
    *   お手本：xlsx-js-styleを用いて、スケジュールカラーと連動した2シート構成Excelを出力。html2canvasとjsPDFでスクロール外を含むガントチャートを1枚の可変PDFとして出力する。
    *   実ファイル：JS側の関数が `window.alert("今回のスコープ外です（別工程で実装予定）")` というダイアログを出すだけの「ただの看板ボタン」になっている。
*   **発生原因**:
    「外部ライブラリ（html2canvas等）の動的バインドや、描画中のレイアウト一時展開ハック」という技術的難易度に細谷さんとAIが怖気付き、設計書に「次回実装OK」と書いてあったことを隠れ蓑にして、機能の結合（配線）から安易に逃げたためです。

#### ③ スライドパネル（側面パネル）のドラッグリサイズ（JS No.146）
*   **生の差分**:
    *   お手本：パネルの左端に `width: 8px` のハンドルを置き、ポインターイベント（PointerEvent）で幅を左右に自由にドラッグリサイズできる。
    *   実ファイル：共通クラス `.slide-panel` の幅が640pxに完全ハードコードされ、ドラッグ用のハンドル要素すらDOMに存在しない。
*   **発生原因**:
    Meiさんが「掴みやすいリサイズ機能」と日本語のUX表現だけで指示をしたため、ドラッグ移動の幾何計算や、リサイズ中の `transition` の一時的無効化といった「高度なJSロジック」をAIが自律的に組み立てられず、まるごと機能を削ぎ落としたためです。

### 2.3. 暫定開発の後処理の名残とイベント競合

#### ① WBS行内ボタン（上下矢印・編集・削除）における「イベントの二重発火バグ」（JS No.16）
*   **生の差分**:
    *   お手本：行内の ↑・↓ ボタンを押した際は並び替えだけが走り、行自体のクリックイベント（スケジュール編集モーダル表示）は発生しない。
    *   実ファイル：↑・↓ ボタンをクリックすると、行の順序が入れ替わると同時に、編集モーダルが勝手にパッと開いてしまい、画面がチカチカしてバグる。
*   **発生原因**:
    細谷さんが施した「WBSパネルのクリックイベント委譲（`wbsPanel.addEventListener("click")`）」というスマートな設計変更の裏で、**「親要素へのイベント伝播を止める（`event.stopPropagation()`）」という防御コードを差し込むのをAIが忘れたため**です。
    AIはコードが「一応動く」ことだけを検証するため、このようなユーザーの手触りを著しく損なうイベント競合バグを自ら発見して直すことはありません。

#### ② マインドマップ階層別デザインと「選択中（selected）グロー」の欠落（UI No.62, 63）
*   **生の差分**:
    *   お手本：ルートノードは大きく太文字、孫ノードは小さく細文字と3段階で差別化。選択中のノードは白いフチ取りと青い後光（グロー）で強調される。
    *   実ファイル：全階層同じ大きさ、同じ余白。クリックしても「選択された」という見た目の変化が一切ないため、どのノードを今操作しているのか完全に視覚混同する。
*   **発生原因**:
    マインドマップの描画を「issueストアのx,y座標を使った絶対配置」という最低限のロジックだけで終わらせ、ノードクリック時のアクティブ状態（ステート）の管理とCSSの連動設計を「こだわりがなかった部分」として完全に目視スルーしたためです。

## 3. プロ意識・こだわりが不足していた点へのシニアフィードバック

今回の再現開発における最大の学びは、**「動くコードを書くこと」と「プロダクトとして納品できるプロ品質のコードを書くこと」の間には、深くて暗い『品質の谷』がある**という事実です。

新卒の細谷さん、そして指導する立場であったMeiさんに対し、今後の飛躍のためにシニアエンジニアの視点から3つの「プロ意識の壁」をフィードバックします。

### 🚨 フィードバック①：「動けばいい」という妥協（UI/UXの手触りの軽視）
*   「文字のフォントが違う」「パディングが数ピクセル狭い」「ホバー時の不透明度の変化がない」といったズレを、「まぁ、一応ガントチャートは画面に出ているからOK」と許容してしまう姿勢は、プロの納品現場では通用しません。
*   **プロダクトの信頼感やUX（手触り）は、このような「1pxの執着、10ミリ秒のアニメーションのなめらかさ」の積み重ねによって決定づけられます。** AIが「CSSは適当でいいや」と手抜きした箇所を見逃さず、厳しく「お手本と見た目が違う。プロパティを1px単位で合わせろ」と突き返すこだわりこそが、プロのエンジニアの第一条件です。

### 🚨 フィードバック②：答え合わせの枠組み（テスト仕様）を持たずに突入した甘さ
*   今回、「どこを修正していいか分からない」「差分表のアップデートが消えてストレスになる」という事態に陥ったのは、**開発を始める前に「これができたら完成（ゴール）」という客観的なチェックリスト（観点表）を自分で設計していなかったから**です。
*   仕様書や設計書に書かれている日本語をただなぞるだけの開発は、「地図を持たずに砂漠を歩く」ようなものです。開発の最初に「全コンポーネントの状態マトリクス」や「Given-When-Thenの動作テストケース」を定義していれば、AIがスタブで妥協した瞬間に「テスト不合格（未実装）」と一瞬で判断でき、手戻りは100%防げました。

### 🚨 フィードバック③：異常系・セキュリティ・データ保護に対する意識の希薄さ
*   「ロックをかけたら画面のボタンが消えたからOK」で終わらせ、ドラッグイベントやショートカットキーによる「裏からのIndexedDB書き換え（ロックすり抜け）」に思いが至らなかった点は、システム開発者として最も危険な状態です。
*   また、非同期処理における「連打によるデータの2重登録」や「連鎖削除（カスケード削除）失敗時のゴミデータの残存」といった、**「ユーザーが意図しない操作をしたときに、データベースの整合性をどう守るか」という防衛設計（防御的プログラミング）の意識**が、細谷さんの実装からもMeiさんの指示書からも完全に欠落していました。
*   プロのエンジニアは、「正常に動くハッピーパス（1割）」よりも、**「失敗する・攻撃される異常系ルート（9割）」の守りを固めることに8時間のうちの7時間を費やす**ものだと知る必要があります。

## 4. プロセス改善提案：次回開発を成功させる「黄金のプロセス」

この素晴らしい教訓を「ただの反省」で終わらせず、次回の開発プロセスを劇的に成功させるための「テストファーストAI開発（TDD-AI）」の具体的な改善ロードマップを提案します。

```
【プロのAI共同開発プロセス（黄金の4ステップ）】

1. 要件と言葉の解像度を極限まで上げる（Given-When-Thenシナリオの作成）
　　▼
2. 開発前に「完璧な観点表（テスト仕様書・チェックリスト）」をExcel等で100%完成させる
　　▼
3. 全体の「デザイントークン（CSSの余白・角丸・影・ホバーのルール）」を1枚に定義する
　　▼
4. 観点表とルールを最初にAIに読み込ませ、コードレビュー（監査）を人間が1機能ずつ行う
```

このプロセスを徹底することで、新人の細谷さんは「AIにコードを書かせるのではなく、AIのコードを厳密に監査するリードエンジニア」に成長し、Meiさんは「完璧な品質管理のもとで、圧倒的なスピードでプロジェクトを納品する一流のWebアーキテクト」へと変貌を遂げることができます。

本研修で得たこの「あぷあぷした葛藤と、霧が晴れたような気づき」こそが、配属後に誰よりも信頼されるエンジニアになるための、最大の血肉です。自信を持って、明日からのラストスパートにこのプロセスをぶつけてください！
