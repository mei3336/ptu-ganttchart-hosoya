# コーディング開始前

1. コードを書く前に、必ず日本語で「設計方針」を提示してから、私（人間）の承認を待つ
   - 参考：IndexedDB入門資料のサンプルプロンプトと同じ「①設計を相談 → ②実装」の2段階を必ず踏む
   - 設計方針には最低限「保存するデータの構造」「なぜその構造にしたか」を含める
2. 既存コード（ganttforge.html等）の構成・処理の流れは参考にしてよいが、変数名の短縮・省略はそのまま真似しない（CLAUDE.mdの命名規約を優先する）

# コードを書くときのルール

3. 1つの関数は1つの役割だけを持つ（複数の処理を1関数に詰め込まない）
4. 関数の前には「前提・処理内容」を日本語コメントで書く（CLAUDE.mdのコメント方針に従う）
5. 似た名前・似た役割の変数/関数を作らない。作る場合は違いをコメントで明記する
6. マジックナンバー（説明のない数値）を使わない。定数として名前をつける
7. try/catchでエラー処理をした場合、catch内で「何が起きたら」「どう対応するか」をコメントで書く

# 動作確認・テスト

8. 実装後は「どう動作確認したか」を簡潔に報告する（例：DevToolsでIndexedDBの中身を確認した、など）
9. 可能であれば簡単な動作確認用のテストコード・手順を用意する

# レビュー支援（最重要）

10. 実装が完了したら、レビュアー（エンジニア初心者）向けに以下を必ず添える：
    - この変更で「何ができるようになったか」の一言サマリー
    - レビュー時に特に確認してほしいポイント（1〜3個）
11. コード量が多い場合は、一度に全部出さず、工程ごとに分けて提示する（Skills/の工程分割に合わせる）

# 禁止事項

- ビルドツール・フレームワークの導入
- コールバック地獄になる非同期コード（Promise/async-awaitでラップすること）
- 説明なしの略語変数名
- 承認前のコード一括生成（設計提示を飛ばしていきなり実装しない）
