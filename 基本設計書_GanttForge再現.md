# 基本設計書：GanttForge再現アプリ

## 0. 本書の位置づけ

- 対象：`ganttforge.html`（単一HTML・IndexedDB想定・ビルドツール不使用）の再現アプリ
- 再現範囲：**（全機能再現）**
- ヒアリング方式：発注者（顧客役）細谷芽生が元アプリを操作し、目視確認した結果をもとに作成
- 未確認・要検証の事項は本書に含めず、別紙「質問リスト_未確認事項.md」にまとめている
- ヒアリングの過程で見つかった「元アプリの既知の非整合（バグの可能性がある挙動）」は、**忠実再現した上で本書に明記する**方針

## 1. 前提条件・注意事項

### 1.1 保存技術についての重要な前置き

`ganttforge.html`のソースコードを確認したところ、**実際の保存先はIndexedDBではなくlocalStorage**である（`_ls()` / `_ss()` 関数がlocalStorageを直接操作）。加えて、初回起動時のみ「旧バージョンのIndexedDB（`ganttforge_v2`という名前のDB）からlocalStorageへの一度きりのデータ移行処理」が存在する。

つまり元アプリは「かつてIndexedDBを使っていたが、その後localStorageに移行したアプリ」という状態にある。

**本プロジェクトでの扱い**：CLAUDE.mdの技術方針（「データの永続化はIndexedDBの生APIを使用する」）を優先し、**保存先はIndexedDBで実装する**。画面仕様・機能仕様は保存技術に依存しないため、この読み替えによる仕様上の影響はない。

### 1.2 用語・ストア名の対応

| UI表示（機能名） | 元アプリの内部ストア名 | 新アプリのストア名 |
|---|---|---|
| スケジュール（ガントチャート側の作業項目） | `tasks` | `schedules`（複数形） |
| タスク管理（進捗管理側の作業項目） | `todos` | `tasks`（複数形） |
| マインドマップのノード | `issues` | `issues`（マインドマップ専用データ。独立した「課題管理」機能は存在しないと確定） |
| プロジェクト | `projects` | `projects` |
| コメント | `comments` | `comments` |
| メモ | `memos` | `memos` |
| 即時メモ | `quickmemos` | `quickmemos` |
| スナップショット | `snapshots` | `snapshots` |
| 変更履歴 | `changelog` | `changelog` |

以降、本書では新アプリのストア名（`schedules`／`tasks`）で統一して記述する。変数名・関数名レベルでも同様の読み替えを行う（例：`saveTask`→`saveSchedule`）。

### 1.3 既知の非整合事項（忠実再現の対象・要注意）

| No | 内容 | 対応方針 |
|---|---|---|
| 1 | **プロジェクト**の終了日を開始日より前に設定しても保存できてしまう（期間3年超のみエラーになる）一方、**スケジュール**では同じ操作をすると「終了日は開始日以降にしてください」とエラーになる。同じ日付整合性チェックなのに、プロジェクトとスケジュールで実装レベルが異なる（確認済み：2026/7/14ヒアリングにて再現） | 忠実再現。プロジェクト側はバリデーション無し、スケジュール側はバリデーション有り、という非対称な仕様のまま実装する |
| 2 | タスク管理（進捗管理側）とスケジュール（ガントチャート側）のステータスは連携していない | 忠実再現。改善案は本書「9. 改善提案（スコープ外）」に記載 |
| 3 | 変更履歴からの「戻す」は、レコード1件を書き戻すだけで親子関係の整合性を保証しない（親のいない子レコードが発生し得る） | 忠実再現。改善案は本書「9. 改善提案（スコープ外）」に記載 |
| 4 | データインポートは1件ごとに変更履歴へログが記録されるが、スナップショット復元は変更履歴に一切記録されない（確認済み）。同じ「一括書き換え」操作なのに扱いが異なる | 忠実再現。改善案は本書「9. 改善提案（スコープ外）」に記載 |

## 2. 画面構成

### 2.1 ヘッダー（2段構成）

**第1ヘッダー（グローバル操作）**

| 左 | 右 |
|---|---|
| GanttForgeロゴ／プロジェクト選択ドロップダウン／＋新規プロジェクト | データ取込／データ出力／PDF出力／Excel出力 |

- データ取込・データ出力：プロジェクト未選択でも使用可
- PDF出力・Excel出力：プロジェクト未選択時は使用不可（活性制御が必要）

**第2ヘッダー（プロジェクト操作バー）**

| 左 | 右 |
|---|---|
| ロック／＋スケジュール追加／表示粒度（日・週・1ヶ月・3ヶ月）／今日／全て開く・全て閉じる／開始・終了・状態列トグル／マイルストーン／編集 | スナップショット／即時メモ／メモ／コメント／変更履歴／マインドマップ／タスク管理 |

### 2.2 各ボタンの機能仕様

| ボタン | 機能 |
|---|---|
| 今日 | ガントチャートを今日の日付位置にスクロール |
| 全て開く／全て閉じる | 押すたびに「開く」「閉じる」を表示し合いながら機能が切り替わる。**個々の行の実際の開閉状態は判定材料にせず、ボタン自体が持つ状態フラグをそのまま反転させるだけの単純な実装** |
| 開始／終了／状態 | スケジュール一覧テーブルに列を追加表示するトグル。表示順は常に「開始→終了→状態」の固定順で、右側から列が追加される |
| マイルストーン | 「マイルストーン管理」モーダルが開く。「マイルストーン名」「日付」の入力行が複数並び、各行に削除用の×ボタンがある。「＋マイルストーンを追加」で行を追加でき、**1回の操作で複数のマイルストーンをまとめて追加・削除できる**。キャンセル／保存ボタンあり（確定） |
| 編集 | プロジェクト編集モーダルが開く（名称・開始日・終了日・マイルストーンを編集） |

### 2.3 共通UI部品：通知パターン

これまでのヒアリングで確認した警告・通知の挙動を横断的に見ると、アプリ全体で以下の2パターンに統一されていることが分かった。

| パターン | 表示位置 | 消え方 | 用途 |
|---|---|---|---|
| 警告ポップアップ（アラート） | 画面中央最上部 | **OKを押すまで消えない** | 入力エラー・削除確認など、ユーザーの判断を止めて対応させたい場面 |
| トースト通知 | 画面右下 | **数秒後（目安3秒）に自動で消える、さりげない表示** | 保存・削除・出力完了など、処理が成功したことを軽く知らせる場面 |

この2パターンは、これまで個別に確認してきた「プロジェクト名未入力の警告」「削除確認ダイアログ」「保存しました」「削除しました」等の通知すべてに共通する設計ルールとして扱ってよい。

**実装上の注意**：トースト通知は「さりげなさ」を重視した表示のため、**動作確認の際に見逃しやすい**。ヒアリングの過程でも、成功通知の有無を明示的に確認していない機能がいくつかある（例：メモの全削除時、マイルストーン追加時など）。実装時・動作確認時は、操作のたびに画面右下を意識して確認すること。



### 3.1 新規作成・編集フォーム

| 項目 | 仕様 |
|---|---|
| プロジェクト名 | テキスト入力、必須 |
| 開始日 | カレンダー入力、デフォルト＝今日 |
| 終了日（最大3年） | カレンダー入力、デフォルト＝今日、上限は開始日から3年後 |

### 3.2 バリデーション

| 条件 | 結果 |
|---|---|
| プロジェクト名が空 | 「プロジェクト名を入力してください」アラート（OK押下まで消えない） |
| 期間が3年超 | 「期間は最大3年です」アラート（OK押下まで消えない） |
| 終了日 < 開始日 | 保存できてしまう（既知の非整合、質問リストで再確認予定） |

### 3.3 プロジェクト切替・削除・ロック

- 切替：ドロップダウン形式
- 削除：確認ダイアログ「『{プロジェクト名}』を削除しますか？（すべてのスケジュールも削除されます）」→OKで配下スケジュール・関連データも全削除
- ロック：トグルON時、ヘッダーとガントの間に薄赤帯＋赤ボーダー＋「🔒ロック中—編集操作は無効です」表示。「スケジュール追加」「マイルストーン」ボタンが淡色化・無効化、プロジェクト削除ボタンが非表示。他機能は操作可能

## 4. スケジュール管理（旧`tasks`→新`schedules`）

### 4.1 階層構造

- 親→子→孫の3階層まで作成可能（ひ孫は作成不可）
- WBS番号を自動採番（1, 1.1, 1.1.1 …）
- 各行の「＋／－」ボタンで個別に開閉可能
- 並び替え（上下矢印）は、**同一階層（同じ親を持つ兄弟グループ）内でのみ**可能。親をまたいだ移動は不可（確定）
- 各グループの先頭行では上矢印、末尾行では下矢印が完全に無効化（淡色表示・クリック無反応）される。この制御は親同士・子同士・孫同士のいずれの階層でも共通（確定）

### 4.2 新規追加・編集フォーム

| 項目 | 仕様 |
|---|---|
| スケジュール名 | テキスト入力、必須 |
| 開始日・終了日 | 子を持たない場合のみ手入力可。子を持つ場合は「子タスクから自動計算」と表示され読み取り専用 |
| 担当者 | テキスト入力 |
| ステータス | 未着手／進行中／完了 |
| カラー | カラーパレットから選択 |
| メモ | 詳細・注意事項用のテキストエリア |
| コメント | 編集モーダル内に埋め込み。既存スケジュールにのみ表示 |

### 4.2.1 バリデーション

- 終了日 < 開始日の場合、「終了日は開始日以降にしてください」とアラート表示され保存不可（**プロジェクトのバリデーションとは異なり、スケジュールにはこのチェックが実装されている**。1.3既知の非整合No.1を参照）

### 4.3 日付の親子連動仕様

- 子の開始日・終了日を変更すると、親の日付は「全子の最早開始日〜最遅終了日」に自動更新される
- 更新は上位階層へも再帰的に伝播する（孫の変更が祖父母まで届く）
- 期間が縮む方向にも追従する
- 子を全て削除した場合、親の日付は「削除直前の同期計算結果」がそのまま残る（再計算・リセットは行われない）

### 4.4 削除

- 親を削除すると、配下の子・孫スケジュールも再帰的に削除される
- 削除されたスケジュールに紐づくコメントも削除される
- 削除後、親（残っている場合）の日付は残存する子の日付に合わせて再計算される

## 5. タスク管理（旧`todos`→新`tasks`）

- バックログ形式のステータス管理（例：backlog／doing／done）
- 項目：タイトル、ステータス、優先度（高・中・低・なし）、期日、メモ（詳細）
- スケジュール（ガントチャート側）とは**データ連携していない**（既知の非整合、対応方針は既知の非整合表を参照）

## 6. マインドマップ

- 「メインテーマ」を起点としたツリー型マインドマップ（ノード同士を曲線で接続）
- **メインテーマの表示内容はプロジェクト名**（固定文字列ではない）。プロジェクトが選択されていない、またはプロジェクト情報を参照できない状態（スナップショット復元・データインポート直後にプロジェクト選択が外れた場合など）になると、フォールバック表示として「メインテーマ」という文字列がそのまま表示される（確認済み）
- 右クリックメニュー：名前を変更（F2）／子ノード追加（Tab）／兄弟ノード追加（Enter）／折りたたみ（Space）／削除（Del）
- メインテーマ（ルートノード）には「兄弟ノード追加」「削除」は表示されない
- ノードは9色のカラーパレットから選択可能
- **確定**：マインドマップのノードは`issues`ストアに保存されている。「課題管理」という独立した機能は存在せず、`issues`という内部ストア名はマインドマップ専用のデータを指すと考えてよい（1.2用語対応表を参照。設計書全体で「issues＝マインドマップのノードデータ」として扱う）

## 8. 変更履歴（changelog）

### 8.1 記録対象・粒度

- 記録対象：プロジェクト／スケジュール／タスク管理の追加・変更・削除
- 「追加」ログには「戻す」ボタンが**表示されない**（新規作成はロールバック対象外）
- 「変更」ログは、1回の保存で複数フィールドが変わった場合も**1つのログにまとめて**記録される（例：「開始日：A→B」「終了日：C→D」を1行に併記）
- 「削除」ログは「削除されました」の表示のみ、戻すボタンあり
- 親を削除した場合、親・子・孫それぞれについて**個別に**削除ログが記録される（3件なら3件のログ）
- 一括処理（データインポート・スナップショット復元等）の際も、特別な集約ログにはならず、**1件保存するたびに1行ログが追加される**単純な仕組みになっている（インポート時の動作確認で確認済み）
- フィルタタブ「すべて／追加／変更／削除」で絞り込み可能
- フッターに全件数（追加／変更／削除の内訳）を表示

### 8.2 「戻す」の挙動と既知の制約

- 「戻す」は該当ストアの該当レコード1件を、ログに保存されている過去の状態でそのまま上書きするだけの処理
- 親子関係の整合性チェックは行われない。そのため、子のみを「戻す」で復元した場合に親が存在しないと、データ上は存在するが画面には表示されない「幽霊レコード」が発生し得る（既知の非整合。対応方針は既知の非整合表を参照）

### 8.3 スナップショット復元・データインポートとログ記録の違い（確定）

- **データインポート（差分適用）**：選択した項目を反映すると、1件ごとに変更履歴へログが記録される（8.1で確認済み）
- **スナップショット復元**：実行しても**変更履歴には一切記録されない**（2026/7/14ヒアリングにて確認済み）。復元処理はログを記録しない低レベルAPIで実装されている可能性が高く、同じ「一括でデータを書き換える」操作でありながら、インポートとは扱いが異なる。忠実再現の対象とする

## 9. データ取込／エクスポート

### 9.1 エクスポート（JSON）

- 対象：projects, schedules(旧tasks), tasks(旧todos), issues, comments, memos, quickmemos, snapshots の全ストア
- ファイル形式：JSON（ファイル名例：`GanttForge_backup_YYYYMMDD.json`）

### 9.2 インポート（JSON）

- ファイルに`projects`・`schedules`キーが無い場合は「無効なファイルです。GanttForgeのエクスポートファイルを選択してください。」とエラー表示
- インポート時は「データ取込－差分比較」モーダルが開き、ストアごとにグループ化して変更・削除候補を一覧表示
- 各項目にチェックボックスがあり、選択した項目のみ「選択した変更を適用」ボタンで反映（全選択チェックボックスあり）
- **確定**：チェックを入れた「削除」項目は、実際に削除が実行される（確認済み）
- 適用前にブラウザ標準のconfirmダイアログで最終確認（「n件の変更を適用しますか？」）
- 適用後は画面右下にトースト通知（「取込完了：新規n件／変更n件／削除n件」）

### 9.3 PDF出力（確定・実物確認済み）

- タイトル：「{プロジェクト名} — ガントチャート」
- 左側：スケジュール一覧（No／階層インデント付きスケジュール名）
- 右側：ガントチャート（今日のマーカー付き、月単位の区切り線）
- **用紙の縦横・サイズは固定ではなく、実際に登録されているスケジュールの最終日ギリギリに収まるよう、右端が可変で出力される**（＝A4等の定型サイズに収める設計ではなく、コンテンツ量に応じた可変サイズのPDFを生成している）
- 右上に出力日時等の情報欄あり（実物では要素が見切れて確認できなかった箇所もあり、細部は別途確認）

### 9.4 Excel出力（確定・実物確認済み）

2シート構成：

**シート1：ガントチャート**

| 行 | 内容 |
|---|---|
| 1行目 | 「{プロジェクト名} — ガントチャート」＋月単位の見出し（結合セル、2026年7月〜数年先まで） |
| 2行目 | 列見出し：No／WBS／スケジュール／階層／開始日／終了日／期間／担当者／ステータス、それ以降は日付（1日単位の連番） |
| 3行目以降 | 各スケジュールのデータ行（階層は「親」「子」等の表記、期間は日数） |
| データ最終行の次 | 「合計（ルート期間合計）」の集計行 |
| 末尾 | 「出力日時：YYYY/M/D H:mm:ss」の行 |

- 日付列は1日1列で、稼働している日にセルの色が付く形式（ガントチャートの罫線をそのままExcelのセル色で再現）と推測される

**シート2：サマリー**

| 項目 | 値の例 |
|---|---|
| プロジェクト名 | 妹の作家活動プロデュース |
| 開始日 | 2026-07-14 |
| 終了日 | 2029-04-25 |
| 表示期間 | 1017日 |
| 総タスク数 | 1件 |

※「終了日」「表示期間」は、プロジェクト自体の終了日ではなく、スケジュールの実効範囲（4.3で確認した「実効日付範囲」の考え方）に基づいている可能性が高い。プロジェクト単体の終了日と食い違う場合があるため、実装時に計算式を要確認（質問リストに追記）

## 10. メモ／即時メモ

| 項目 | メモ | 即時メモ |
|---|---|---|
| コンセプト | 複数ファイルを左に並べて管理するエディタ（VSCode簡易版のイメージ） | 時系列の書き捨てメモ（Teamsの自分宛チャットのイメージ） |
| 構成 | 左：メモ一覧（タイトル＋更新日時）／右：編集エリア | 単一タイムライン＋下部入力欄 |
| モード | 編集／プレビュー／分割の3タブ、Markdown対応 | 常に時系列リスト＋入力欄のみ、Markdown対応 |
| 新規作成 | 「＋新規メモ」で複数ファイルとして管理 | 「投稿」ボタン、または`Ctrl+Enter`で追加 |
| 個別削除 | ゴミ箱アイコン（確認ダイアログあり「『{タイトル}』を削除しますか？」→削除後トースト「メモを削除しました」） | ホバー時ゴミ箱アイコン→**1クリックで即削除、確認ダイアログ・成功通知なし** |
| 編集 | 可能 | **不可**（削除して投稿し直すのみ） |
| 全削除 | なし（複数メモをファイルとして個別管理するため） | あり（「全削除」ボタン、確認ダイアログあり） |

### 10.1 補足：即時メモの削除確認なしについて

即時メモの「確認なしの即削除」は、他の削除操作（確認ダイアログあり）と一貫していない点だが、「書き捨てメモ」という機能の性質上、仕様として扱う（既知の非整合表には含めない）。

### 10.2 メモ：新規作成時のタイトル引き継ぎバグ（原因判明・忠実再現しない）

**現象**：直前に編集していたメモのタイトルが、そのすぐ後に作成した新規メモのタイトルとして引き継がれてしまう場合がある（発生は不安定）。

**原因（コード確認済み）**：タイトル・本文の保存は、入力停止から600ミリ秒後に自動実行される仕組みになっている。「＋新規メモ」を押す処理は、この保留中の自動保存が完了する前に「現在選択中のメモ」を新規メモに切り替えてしまうため、直前のメモ向けだったはずの保留中の保存が、切り替え後の新規メモに対して実行されてしまう。直前の編集から時間が経っていて保留中の保存が残っていなければ発生しない（別画面に移動してから戻ると発生しないのはこのため）。

**対応方針**：これは仕様ではなく明確な実装ミス（保存対象の切り替えタイミングの誤り）であるため、**竹方式（忠実再現＋備考記録）ではなく、松方式（再現せず正しく実装する）を採用**する。新規メモ作成時は、直前のメモの保留中の保存を先に確定させてから新規メモに切り替える、という順序で実装すること。
