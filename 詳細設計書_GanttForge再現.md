# 詳細設計書：GanttForge再現アプリ

## 0. 本書の位置づけ

- 上位文書：`基本設計書_GanttForge再現.md`（画面仕様・データ設計の確定内容）
- 参照ソース：`ganttforge.html`（既存アプリの実装。以降「元アプリ」と呼ぶ）
- 本書の目的：各機能の**処理内容（正常系の流れ）**と、**エラーバリエーション・通知内容（異常系／成功時）**を、機能単位で網羅的に整理すること
- 記述方針：元アプリのソースコードから読み取れる挙動を**忠実に**書き出す。基本設計書で「既知の非整合」として明記済みの事項は、本書でも同じ非対称のまま記載する（勝手に統一しない）
- 未確認・要判断の事項は本書に含めず、別紙「質問リスト_詳細設計未確認事項.md」にまとめる
- 各項目には元アプリの該当関数名・行番号を `(参照: 関数名 L行)` の形式で付記し、実装時にソースへ立ち返れるようにする

## 1. システム構成・技術スタック

### 1.1 全体構成

```
┌─────────────────────────────────────────────┐
│  ブラウザ（Chrome等）                         │
│                                               │
│  ┌─────────────────────────────────────┐    │
│  │  index.html（単一ファイル）           │    │
│  │  ─────────────────────────           │    │
│  │  ・HTML（画面構造）                   │    │
│  │  ・CSS（style内・ビルド不使用）        │    │
│  │  ・JavaScript（生API・async/await）   │    │
│  └───────────────┬───────────────────┘    │
│                  │                           │
│                  ▼                           │
│  ┌─────────────────────────────────────┐    │
│  │  IndexedDB（ブラウザ内蔵DB）           │    │
│  │  ストア：projects / schedules /       │    │
│  │  tasks / issues / comments / memos /  │    │
│  │  quickmemos / snapshots / changelog   │    │
│  └─────────────────────────────────────┘    │
└─────────────────────────────────────────────┘
        │                          │
        ▼                          ▼
  （JSON手動出力/取込）      （PDF/Excel手動出力・保存はしない）
```

- ビルドツール・フレームワーク不使用。単一HTMLファイルをブラウザで直接開く、またはローカルサーバー（`localhost`）で開く方式（CLAUDE.md準拠）
- サーバーサイド無し。データはすべてブラウザのIndexedDBに閉じる（他端末・他ブラウザとは共有されない）
- 元アプリにあった `serviceWorker` 登録処理（`sw.js`）は本再現の**対象外**とする（`sw.js`の実体が今回の参照ファイル一式に含まれておらず、オフライン対応（PWA化）は研修の主目的である「IndexedDBでのCRUD実装力・設計文書化力」とは別テーマのため。方針確定）

### 1.2 使用ライブラリ（CDN経由・元アプリと同一を踏襲）

| ライブラリ | バージョン | 用途 | 読み込み元 |
|---|---|---|---|
| html2canvas | 1.4.1 | PDF出力時に画面をキャプチャして画像化 | `cdnjs.cloudflare.com` |
| jsPDF | 2.5.1 | キャプチャ画像からPDFファイルを生成 | `cdnjs.cloudflare.com` |
| xlsx-js-style | 1.2.0 | Excelファイル（スタイル付き）の生成 | `cdn.jsdelivr.net` |

- 上記3点はCLAUDE.mdの「外部ライブラリはCDN経由でのみ利用可」に合致するため、そのまま採用する
- ビルド工程が無いため、CDNの読み込み失敗（オフライン等）を検知する処理が必要（後述 8.4 参照）

### 1.3 データ永続化方式（基本設計書1.1の再掲・重要）

- 元アプリの実際の保存先は **localStorage**（`_ls()` / `_ss()` 関数が直接操作）であり、IndexedDBではない
- 本プロジェクトではCLAUDE.mdの技術方針に従い、**IndexedDBの生API**で実装する
- 元アプリの `dbGetAll` / `dbPut` / `dbDel` 等の関数名はそのまま踏襲しつつ、内部実装のみをlocalStorage操作からIndexedDBのトランザクション処理に置き換える方針とする
- `_dbPutRaw` / `_dbDelRaw`（変更履歴に記録しない低レベルAPI。スナップショット復元で使用）も同様にIndexedDB版として用意する

### 1.4 非同期処理の方針（CLAUDE.md準拠）

- すべてのDB操作はPromiseでラップし、呼び出し側は `async/await` で記述する
- IndexedDBのトランザクションは自動コミットされるため、トランザクションを開いてから `fetch` 等の他の非同期処理を挟まない（挟むとトランザクションが閉じてしまう）
- 複数ストアにまたがる書き込み（例：プロジェクト削除時の配下スケジュール一括削除）は、先に `getAll` 等で対象データを読み切ってから、書き込み用トランザクションを開始する

## 2. 共通仕様：通知パターン（基本設計書2.3の詳細化）

全機能に共通する通知の実装ルールを以下に統一する。

| パターン | 実装方法（元アプリ） | 表示位置・消え方 | 用途 |
|---|---|---|---|
| 警告アラート | `alert(message)` | 画面中央最上部にブラウザ標準ダイアログ、**OK押下まで消えない** | 入力エラー・操作不能の通知 |
| 確認ダイアログ | `confirm(message)` | ブラウザ標準ダイアログ、OK/キャンセル選択式 | 削除など取り消せない操作の事前確認 |
| トースト通知 | `showToast(message)` | 画面右下、**約2.8秒後に自動で消える**（`setTimeout` 2800ms、参照: `showToast` L4170-4174） | 処理成功の軽い通知 |

- `showToast` は要素`#toast`に`show`クラスを付与し、`CSSトランジション`でスライドイン表示する。連続してトーストを呼ぶと`clearTimeout`で前のタイマーをリセットし、最後の呼び出しから2.8秒後に消える（参照: L4170-4174）
- 本再現アプリでもこの2パターン（`alert`/`confirm`によるブロッキング通知、`showToast`による非ブロッキング通知）をそのまま踏襲する

## 3. 機能別詳細設計

### 3.1 プロジェクト管理

#### 3.1.1 新規作成・編集（`submitProject` L2298-2313）

**処理内容（正常系）**
1. フォームから プロジェクト名・開始日・終了日 を取得
2. 下記バリデーションを順に実行（すべて通過した場合のみ次へ）
3. 新規の場合は `genId()` でID発番、編集の場合は既存IDを維持
4. `dbPut('projects', p)` で保存
5. `loadProjects()` でプロジェクト一覧を再読込
6. プロジェクト選択欄（`projectSelect`）に保存したプロジェクトを選択状態にする
7. `selectProject(p.id)` でガントチャート画面を再描画
8. トースト「プロジェクトを保存しました」を表示

**バリデーション（エラーバリエーション）**

| No | 条件 | 通知内容 | 種別 | 参照 |
|---|---|---|---|---|
| 1 | プロジェクト名が空文字 | 「プロジェクト名を入力してください」 | alert | L2301 |
| 2 | 開始日または終了日が未入力 | 「日付を入力してください」 | alert | L2302 |
| 3 | 終了日 ≦ 開始日 | 「終了日は開始日より後にしてください」 | alert | L2303 |
| 4 | 期間が3年（365×3+1日）を超える | 「期間は最大3年です」 | alert | L2304 |

- **既知の非整合（基本設計書1.3-No.1）**：上記No.3は`<=`判定のため「終了日＝開始日」も弾かれる。一方、後述3.2のスケジュールでは`<`判定（終了日＝開始日は許容）であり、判定の境界がプロジェクトとスケジュールで異なる。忠実再現の対象とし、統一しない

#### 3.1.2 削除（`deleteCurrentProject` L1522-1535）

**処理内容**
1. `confirm('「{プロジェクト名}」を削除しますか？（すべてのタスクも削除されます）')` で確認（参照: L1525。文言中は「タスク」表記だが、実際に削除されるのは配下の`schedules`データ）
2. OKの場合、配下の全スケジュール・関連データ（コメント等）を削除してからプロジェクト自体を削除
3. `loadProjects()` で一覧再読込
4. トースト「プロジェクトを削除しました」を表示（L1531）

**エラーバリエーション**：削除確認でキャンセルした場合は何もせず処理終了（通知なし）

#### 3.1.3 切替・ロック

| 機能 | 処理内容 | 通知 |
|---|---|---|
| プロジェクト切替 | ドロップダウン選択時に `selectProject(v)` を呼び、`localStorage`に選択中IDを保存（`gf_pid`）、ガントチャートを再描画 | 通知なし |
| ロックON/OFF切替（`toggleLock` L2204-2210） | `S.locked` を反転し `updateLockUI()` でボタン活性・帯表示を更新 | トースト「ロックしました」／「ロックを解除しました」（L2209） |
| ロック中の編集操作 | 「スケジュール追加」「マイルストーン」ボタン等が無効化。個別関数側でも `isLocked()` チェックあり | トースト「ロック中のため編集できません」（L2315, L2522等） |

### 3.2 スケジュール管理（旧tasks・ガントチャート本体）

#### 3.2.1 新規作成・編集（`submitTask` L2377-2401想定範囲）

**処理内容**
1. モーダルのフォームから 名前・開始日・終了日・担当者・ステータス・メモ・カラー を取得
2. 子タスクを持つ親行の場合は日付欄が読み取り専用（`isParentWithChildren`。子の日付範囲から自動算出されるため入力させない）
3. バリデーション通過後、`dbPut('tasks', task)` で保存（新アプリでは`schedules`ストア）
4. 親を持つ場合は `syncAncestorDates(parentId)` で祖先の日付を再計算（後述3.2.3）
5. `renderGantt()` で再描画、トースト「保存しました」（L1567想定）

**バリデーション（エラーバリエーション）**

| No | 条件 | 通知内容 | 種別 | 参照 |
|---|---|---|---|---|
| 1 | 名前が空文字 | 「スケジュールを入力してください」 | alert | L2385 |
| 2 | 開始日または終了日が未入力 | 「日付を入力してください」 | alert | L2387 |
| 3 | 終了日 ＜ 開始日 | 「終了日は開始日以降にしてください」 | alert | L2388 |

- **既知の非整合（基本設計書1.3-No.1・再掲）**：上記No.3は`<`判定のため「終了日＝開始日」は許容される。プロジェクト側（3.1.1-No.3）は`<=`判定で終了日＝開始日を拒否しており、判定基準が非対称。忠実再現とする

#### 3.2.2 削除（`deleteTask` L1569-1590）

**処理内容**
1. `confirmDelete(id)` で確認メッセージを組み立て、`confirm()`表示（実機確認済み・L2265-2269）
   - 子タスクを持つ場合：「「{スケジュール名}」とそのサブタスクをすべて削除しますか？」
   - 子タスクを持たない場合：「「{スケジュール名}」を削除しますか？」
2. OKの場合、当該行および子孫（`getDescendantIds`）をすべて削除
3. 親がいる場合は `syncAncestorDates` で親の日付を再計算
4. `renderGantt()`、トースト「削除しました」（L1589）

**エラーバリエーション**：確認でキャンセルした場合は処理中断、通知なし

#### 3.2.3 日付自動伝播（`syncParentDates` L1536, `syncAncestorDates` L1542, `recalcParentDates` L2082）

**処理内容（正常系のみ・エラー通知なし）**
1. 子タスクの開始日・終了日が変わるたびに、親タスクの開始日・終了日を「子の最小開始日～最大終了日」に自動で再計算する
2. 祖先が複数階層ある場合は `syncAncestorDates` で再帰的に上位へ伝播させる
3. 親行の日付欄はUI上読み取り専用として表示され、`.par-auto-tag`（自動算出タグ）を付与する

- この処理系にはユーザー向けの成功／失敗通知は存在しない（バックグラウンドでの整合性維持処理のため）。処理自体が失敗するケース（DB書き込み失敗等）についてはIndexedDB化に伴い新規にエラーハンドリングを追加する必要がある（後述8章参照）

#### 3.2.4 並び替え・親子変更（ドラッグ操作）

| 機能 | 処理内容 | 通知 |
|---|---|---|
| 兄弟間の並び替え（`moveTask` L2036, `reorderTasks` L2053） | `order`フィールドを更新し保存 | トースト「並び順を変更しました」（L2065） |
| 子タスクへの移動（`reparentTask` L2067） | `parentId`を更新し、旧親・新親双方の日付を再計算 | トースト「子タスクに移動しました」（L2080） |
| バードラッグでの日程変更（`startBarDrag` L2097〜`endBarDrag` L2132-2167） | ガント上のバーをドラッグして開始日・終了日を変更、離した時点で保存 | トースト「スケジュールを更新しました」（L2162） |

**バードラッグの詳細仕様（実機確認済み）**

- ドラッグには3つのモードがある（`bar-lh`/`bar-rh`ハンドルの`pointerdown`ハンドラで指定、L1941-1950）
  - `move`：バー中央をつかんだ場合。開始日・終了日を同じ日数だけ平行移動（期間は変わらない）
  - `resize-start`：左端の掴み手（`bar-lh`）をつかんだ場合。開始日のみ変更
  - `resize`：右端の掴み手（`bar-rh`）をつかんだ場合。終了日のみ変更
- **掴み手（`bar-lh`/`bar-rh`）は「子タスクを持たない行（葉ノード）」にしか付与されない**（`if(!isPar){...}` L1941-1947）。子を持つ親行はこのブロックがスキップされるため掴み手が存在せず、**親行は移動のみ可能でリサイズ（幅変更）はできない**。親の日付は子タスクから自動算出される仕様（3.2.3）のため、意図的にリサイズ不可としていると考えられる
- 掴み手は常時表示ではなく、バーにマウスを乗せた時のみうっすら色が付く（`.gbar:hover .bar-rh,.gbar:hover .bar-lh{background:rgba(0,0,0,.18)}` L297）。バー端のごく狭い領域のみが掴み手判定のため、意図せず`move`扱いになりやすい
- **境界値の丸め処理（実機確認済み・アラート無し）**：範囲外にドラッグした場合、`alert`等の通知は一切出ず、以下の通り黙ってクランプされる（L2109-2131, 2144-2159）
  - `move`：新しい開始日がプロジェクト開始日より前になる場合、期間を保ったままプロジェクト開始日にスナップ。新しい終了日がプロジェクト終了日を超える場合はプロジェクト終了日でカット
  - `resize-start`：新しい開始日が終了日を超える場合は終了日と同日にクランプ。プロジェクト開始日より前にはできない
  - `resize`：新しい終了日が開始日を下回る場合は開始日と同日にクランプ。プロジェクト終了日を超えることはできない
- リサイズ確定時（`endBarDrag`）、子タスクの日付が変わると`syncParentDates`→`syncAncestorDates`が呼ばれ、祖先すべての日付が「その時点の全子タスクの最小開始日〜最大終了日」で**都度上書き**される（3.2.3参照）。親自身はリサイズできないため、親の日付は常に子タスクの変更に追従する一方通行の関係になる

### 3.3 タスク管理（旧todos・進捗管理パネル）

#### 3.3.1 新規作成・編集（`submitTodo` L2994, `saveTodoEdit` L3052）

**処理内容**

2. バリデーション通過後 `dbPut('todos', todo)`（新アプリでは`tasks`ストア）
3. `renderTodos()` で一覧再描画

**エラーバリエーション**

| No | 条件 | 通知内容 | 種別 | 参照 |

#### 3.3.2 削除・一括削除（`deleteTodo` L3009, `clearDoneTodos` L3013-3019）

| 機能 | 処理内容 | 通知（正常系） | エラーバリエーション |

### 3.4 コメント機能

#### 3.4.1 投稿・削除（`postComment` L2420, `deleteComment` L2429／ポップアップ版 `postPopupComment` L2506, `deletePopupComment` L2513）

**処理内容**
1. スケジュール詳細画面またはガント上のポップアップから、コメント本文を入力し投稿
2. `dbPut('comments', comment)` で保存。投稿者情報は保持しない（基本設計書11.5に既定）
3. `loadComments(taskId)` / `loadPopupComments(taskId)` で該当スケジュールのコメント一覧を再取得・再描画

**エラーバリエーション**：**実機確認済み：投稿・削除いずれも成功トーストなし**。空文字投稿時のバリデーション有無、削除時の確認ダイアログ有無についても`alert`/`confirm`の呼び出しは見当たらない

#### 3.4.2 コメント一覧パネル（`loadCommentList` L3531）

- 全スケジュール横断のコメント一覧をスケジュール名でグループ化して表示。グループ単位の「閉じる」操作はUI上の表示制御のみで、データ項目としては保持しない（基本設計書11.5に既定）

### 3.5 マインドマップ（issues・課題管理ではなくノードデータ専用）

#### 3.5.1 ノード追加（`mmAddChild` L3291, `mmAddSibling` L3305）

**処理内容**
1. 選択中ノードの子または兄弟として新規ノードを追加（`title`は仮タイトルで作成後、即座に編集モードへ）
2. `dbPut('issues', node)` で保存
3. `mmAutoLayout()` で座標を自動再計算し `renderMindmap()` で再描画

#### 3.5.2 編集（`mmEditNode` L3318, `mmSaveEdit` L3335）

- タイトルのインライン編集。空文字での保存時の挙動（バリデーション有無）は要質問

#### 3.5.3 削除（`mmDeleteNode` L3379-3390）

**処理内容**
1. 対象ノードとその子孫ノードIDを収集（`toDelete`配列）
2. 収集件数が3件を超える場合のみ確認ダイアログを表示：「{n}個のノードを削除しますか？」（L3383）
3. 3件以下の場合は確認なしで即削除される

**エラーバリエーション（実機確認済み）**

| No | 条件 | 通知内容 | 種別 |
|---|---|---|---|
| 1 | 削除対象（自身+子孫）が3件超（4件以上） | 「{n}個のノードを削除しますか？」 | confirm |
| 2 | 3件以下 | 確認なし・即削除 | ― |

### 3.6 変更履歴（changelog）

#### 3.6.1 記録処理（`writeLog` L1405-1420）

**処理内容（前提・処理）**
1. 対象ストアが `LOGGED_STORES`（projects/schedules/tasks/issues の4種、基本設計書データモデル設計に既定）に含まれる場合のみ記録
2. `action`（add/edit/delete）、`store`、`itemId`、`itemName`、`changes`（`diffFields`で算出したフィールド差分配列、L1396-1404）、`snapshot`（巻き戻し用の直前データ）を1件のログとして保存
3. 通常のCRUD操作（`dbPut`/`dbDel`）を経由した変更はすべて自動的にログ記録される

**基本設計書1.3-No.4の再掲・重要な非対称仕様**
- データインポート（3.9）は1件ごとにこの`writeLog`を経由するため、インポートで反映された全件がログに残る
- 一方、スナップショット復元（3.8.3）は `_dbPutRaw`/`_dbDelRaw`（ログを記録しない低レベルAPI）を使用するため、**復元操作自体は変更履歴に一切残らない**
- 同じ「一括書き換え」でありながら扱いが異なる。忠実再現の対象（改善案はスコープ外）

#### 3.6.2 巻き戻し（`restoreFromLog` L3489-3512）

**処理内容**
1. 対象ログの`snapshot`（変更前のデータ）を取得
2. `confirm('「{名前}」({種別}) をこの時点の状態に戻しますか？')`（L3495）
3. OKの場合、そのレコード1件のみを書き戻す

**エラーバリエーション・注意点**

| No | 条件 | 通知内容 | 種別 |
|---|---|---|---|
| 1 | 復元対象のsnapshotデータが存在しない | 「復元データがありません」 | showToast（L3492） |
| 2 | 確認ダイアログでキャンセル | 処理中断 | ― |
| 3 | （成功時） | 「「{名前}」を復元しました」 | showToast（L3511） |

- **基本設計書1.3-No.3の再掲**：この巻き戻しは1件のレコードを書き戻すのみで、親子関係の整合性（親が既に削除されている場合等）を保証しない。忠実再現とする

#### 3.6.3 履歴の全削除（`clearChangelog` L3513-3519）

| 条件 | 通知内容 | 種別 |
|---|---|---|
| ログが0件 | 「履歴がありません」 | showToast（L3514） |
| ログが1件以上の場合、削除実行前 | 「{n}件の変更履歴を削除しますか？」 | confirm（L3515） |
| 確認OK | 「変更履歴を削除しました」 | showToast（L3518） |
| 確認キャンセル | 処理中断 | ― |

### 3.7 メモ機能

#### 3.7.1 作成・自動保存（`createMemo` L3637, `scheduleMemoSave`/`flushMemoSave` L3668-3674, `saveMemo` L3675）

## 4. 横断的なエラーハンドリング方針（IndexedDB化に伴う追加実装・方針確定）

元アプリはlocalStorage実装のため同期的で例外が起きにくい構造だが、IndexedDBは非同期・トランザクション制御が絡むため、元アプリには存在しない失敗パターンが発生しうる。**IndexedDBを採用する以上、以下のエラーハンドリングは本再現アプリでも標準的な対応として実装する方針とする。**

| No | 想定される事象 | 実装する通知 | 実装方針 |
|---|---|---|---|
| 1 | `indexedDB.open`自体が失敗（ブラウザのプライベートモード制限、ストレージ容量超過等） | 画面初期表示時にalertで「データベースを開けませんでした。ブラウザの設定をご確認ください」等 | `initDB()`相当の初期化処理内で`onerror`をハンドリングし、以降の画面描画を行わずエラー表示に留める |
| 2 | トランザクション中の書き込み失敗（`onerror`） | 各機能の保存処理内で`try/catch`し、「保存に失敗しました」等のトースト＋console.error | 3.8.1の即時メモの実装（`postQuickMemo` L3778）と同じパターン（`try/catch`＋`showToast`＋`console.error`）を、`dbPut`/`dbDel`を呼び出すすべての機能に横展開する |
| 3 | `onblocked`（他タブでDBバージョンが古いまま開かれている） | 「他のタブでこのアプリが開かれています。閉じてから再読み込みしてください」等のalert | `indexedDB.open`の`onblocked`イベントで検知し、alertで案内する |

- 上記は元アプリの忠実再現の範囲を超える**新規追加**だが、「データの永続化はIndexedDBの生APIを使用する」というCLAUDE.mdの技術方針を採用した以上、IndexedDB利用時に一般的に求められる最低限のエラーハンドリングとして実装する
- 実装の優先順位としては、No.1（DB起動失敗）を最優先とする（起動できなければアプリ自体が使えないため）。No.2（個別の書き込み失敗）は`dbPut`/`dbDel`の共通ラッパー関数内で一元的にハンドリングする設計とし、各機能の呼び出し側で個別に`try/catch`を書く手間を減らす。No.3（`onblocked`）は複数タブを同時に開く運用が想定される場合に対応する
