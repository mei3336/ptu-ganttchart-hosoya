# シーケンス図集

基本設計書で確定した仕様のうち、処理の流れが複雑なものをシーケンス図にしています。Mermaid記法なので、対応するビューア（VSCode拡張、GitHub、Mermaid Live Editor等）でそのまま表示できます。

---

## 1. 日付の親子連動（子の変更→親へ伝播）

子スケジュールの日付を変更したときに、親・祖父母まで日付が伝播する流れです。

```mermaid
sequenceDiagram
    actor User as ユーザー
    participant UI as 編集モーダル
    participant Save as saveSchedule()
    participant Sync as syncParentSchedule()
    participant SyncAll as syncAncestorSchedules()
    participant DB as IndexedDB(schedules)

    User->>UI: 子スケジュールの開始日・終了日を変更して保存
    UI->>Save: submitSchedule(子ID, 新しい日付)
    Save->>DB: put(子スケジュール)
    Save->>Sync: syncParentSchedule(子ID)
    Sync->>Sync: 子の親IDを取得
    Sync->>SyncAll: syncAncestorSchedules(親ID)
    SyncAll->>DB: 親の全子スケジュールを取得
    SyncAll->>SyncAll: 最早開始日・最遅終了日を再計算
    alt 親の日付が変化した場合
        SyncAll->>DB: put(親スケジュール、新しい日付)
        SyncAll->>SyncAll: 親に祖父母がいるか確認
        opt 祖父母が存在する
            SyncAll->>SyncAll: syncAncestorSchedules(祖父母ID) を再帰呼び出し
        end
    else 変化なし
        SyncAll-->>Sync: 何もしない（再帰を打ち切る）
    end
    Save-->>UI: ガントチャート再描画
    UI-->>User: 画面に反映
```

## 2. 変更履歴（changelog）の記録

スケジュール・タスク・プロジェクトを保存/削除するたびに、変更履歴が1件ずつ記録される流れです。

```mermaid
sequenceDiagram
    actor User as ユーザー
    participant UI as 編集モーダル
    participant DBPut as dbPut(store, obj)
    participant Diff as diffFields()
    participant Log as writeLog()
    participant ChangelogDB as IndexedDB(changelog)

    User->>UI: スケジュール（または他対象）を編集して保存
    UI->>DBPut: dbPut('schedules', 編集後オブジェクト)
    DBPut->>DBPut: 保存前の旧データを取得
    DBPut->>DBPut: 本体ストアに新データをput
    alt 記録対象ストア（projects/schedules/tasks/issues）である
        alt 新規追加（旧データが無い）
            DBPut->>Log: writeLog('add', ...)
            Log->>ChangelogDB: 追加ログを1件保存（戻すボタンなし）
        else 既存の更新
            DBPut->>Diff: diffFields(旧データ, 新データ)
            Diff-->>DBPut: 変更フィールド一覧（複数可）
            alt 変更が1件以上ある
                DBPut->>Log: writeLog('edit', 変更フィールド一覧)
                Log->>ChangelogDB: 変更ログを1件保存（全フィールドまとめて1行、戻すボタンあり）
            else 変更なし
                DBPut-->>UI: ログ記録なし
            end
        end
    end
    DBPut-->>UI: 保存完了
```

## 3. 親スケジュール削除（カスケード削除とログ）

親を削除すると、子・孫も再帰的に削除され、それぞれ個別にログが記録される流れです。

```mermaid
sequenceDiagram
    actor User as ユーザー
    participant UI as スケジュール一覧
    participant Del as deleteSchedule(親ID)
    participant DB as IndexedDB(schedules)
    participant CommentsDB as IndexedDB(comments)
    participant Log as writeLog()
    participant Sync as syncAncestorSchedules()

    User->>UI: 親スケジュールの削除ボタンを押す
    UI->>Del: deleteSchedule(親ID)
    Del->>Del: 削除対象を再帰的に収集（親→子→孫）
    loop 収集した各スケジュールIDについて
        Del->>DB: dbDel('schedules', 対象ID)
        DB->>Log: writeLog('delete', 対象ID) を個別に記録
        Del->>CommentsDB: 紐づくコメントも削除
    end
    Del->>Del: 削除した親の親ID（祖父母）を取得
    opt 祖父母が存在する
        Del->>Sync: syncAncestorSchedules(祖父母ID)
        Sync->>DB: 残っている兄弟スケジュールから日付を再計算
        Sync->>DB: 祖父母の日付を更新
    end
    Del-->>UI: ガントチャート再描画
    UI-->>User: 削除完了トースト表示
```

## 4. データインポート（差分適用）

JSONファイルを取り込み、選択した差分だけを反映する流れです。

```mermaid
sequenceDiagram
    actor User as ユーザー
    participant UI as データ取込ボタン
    participant Reader as FileReader
    participant DiffCalc as showImportDiff()
    participant Modal as 差分比較モーダル
    participant Apply as applySelectedDiffs()
    participant DB as IndexedDB(各ストア)
    participant Log as writeLog()
    participant Toast as トースト通知

    User->>UI: JSONファイルを選択
    UI->>Reader: ファイル読み込み
    Reader-->>UI: JSON文字列
    UI->>UI: projects/schedulesキーの有無チェック
    alt 不正なファイル
        UI-->>User: 「無効なファイルです」アラート
    else 正常なファイル
        UI->>DiffCalc: showImportDiff(取込データ)
        DiffCalc->>DB: 現在の全ストアデータを取得
        DiffCalc->>DiffCalc: ストアごとに新規/変更/削除候補を算出
        DiffCalc->>Modal: 差分一覧を表示（チェックボックス付き）
        User->>Modal: 反映したい項目にチェックを入れる
        User->>Modal: 「選択した変更を適用」を押す
        Modal->>User: confirm('n件の変更を適用しますか？')
        User->>Modal: OK
        Modal->>Apply: 選択された項目のみ渡す
        loop 選択された各項目
            Apply->>DB: dbPut または dbDel を実行
            DB->>Log: writeLog() で1件ずつログ記録（一括操作の特別扱いなし）
        end
        Apply->>Toast: 「取込完了：新規n件／変更n件／削除n件」
        Toast-->>User: 画面右下に表示
    end
```

## 5. スナップショット復元

保存済みスナップショットから、プロジェクトの状態を復元する流れです。

```mermaid
sequenceDiagram
    actor User as ユーザー
    participant Panel as スナップショットパネル
    participant Restore as restoreSnapshot(snapId)
    participant DB as IndexedDB(schedules)
    participant ProjDB as IndexedDB(projects)

    User->>Panel: 「この状態に復元」を押す
    Panel->>User: confirm('復元しますか？現在のスケジュールは上書きされます')
    User->>Panel: OK
    Panel->>Restore: restoreSnapshot(snapId)
    Restore->>DB: 現在のプロジェクトの全スケジュールを取得
    loop 現在の各スケジュール
        Restore->>DB: dbDel（低レベルAPIで削除、ログ記録なし）
    end
    loop スナップショット内の各スケジュール
        Restore->>DB: dbPut（低レベルAPIで復元、ログ記録なし）
    end
    Restore->>ProjDB: プロジェクトの名称・日付・マイルストーンを復元
    Restore-->>User: 「{名前}に復元しました」トースト表示
    Note over Restore,DB: 復元処理自体は変更履歴に記録されない可能性が高い（低レベルAPI _dbPutRaw/_dbDelRaw を使用）。<br/>データ取込との挙動差異として質問リストで再確認予定
```

**補足**：スナップショット復元は`_dbPutRaw` / `_dbDelRaw`という「ログを記録しない低レベルAPI」を使っている可能性があり、これは「4. データインポート」で確認した「一括処理でも1件ずつログが記録される」という挙動と矛盾するかもしれません。この点は質問リストに追記して再確認することをお勧めします。
